Introduction:

This was a team project, consisting of the following team members: Leonard Holman (CS), Bahar Aybar (MFAC), Deepak Vembar (CS) and myself (ECE). Project 3: Bungee Jumping


This is a description of my team's work on the final project for CPSC 611 - Virtual Reality Systems. We designed a virtual environment using OpenGL comprising of mountains, rivers and valleys that anybody can explore using the head mounted display. The main attractions of our virtual environment are a fly-by through the environment initiated at the press of a button and a bungee jump from a tall bridge above the river. We also have a nice little waterfall right next to the bridge.

The following are the list of team members, along with the credits.

Team:
  • Leonard Holman (CS): Pneumatic chair, stereo vision, Maya .obj object file reader in C++.
  • Deepak Vembar (CS): Waterfall system, fog
  • Bahar Aybar (MFAC): Design of environment in Maya, textures.
  • Vilas Chitrakaran (ECE): HMD1 position/orientation tracking, bungee jump dynamics, audio using OpenAL
Comments:

Our code has been tested on Red Hat Linux 8.0 and requires OpenAL and Glut libraries to compile. We wrote our own code for all modules. Besides OpenAL, no third party non-standard library was used in the development of the project. While Bahar developed the environment (using Maya) as well as the right textures, Leonard hacked away on his object file reader until it actually gave better performance than the ones available freely as open-source. Deepak coded up the waterfall using particles; more than 200 particles are generated per second to create the waterfall. I developed a 1D dynamic mathematical model for the bungee jumper, modelling the jumper and the rope as a nonlinear mass-spring-damper system.

We faced some challenges during the implementation of this project. Besides dealing with the idiosyncrasies of the pneumatic chair and the Flock-of-Birds position tracking hardware, the primary concern was the seamless integration of data from the HMD position tracker with the position/orientation data generated by the sources of movement in the software (fly-by/jump/mouse). In order to easily account for the movement of the user's head (VR display helmet) for all cases, we had to abandon the more stable position/matrix output from the bird for the not-so-stable position/angle output format.

The other major challenge was improving the frame-rate so that the fly-by and the jump felt realistic. The following are some steps we took in order to improve the frame-rate:

  1. Vector operations and trigonometric function calls happen only once every display cycle.
  2. The display callback and the scene update function form a tight loop.
  3. Redundant calls to OpenGL state changes were eliminated.
  4. Reduced the number of function calls by integrating many of them together into one.
  5. The rendering of waterfall is the most computationally expensive component in the environment and in order to avoid performance penalties, the waterfall is switched off whenever the user is not looking at it.
With the above steps, we were able to improve the frame-rate from an unacceptable 3-4 fps to more than 15 fps on View2.
